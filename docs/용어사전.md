# 유니티와 Claude Code 용어 안내서

> 이 문서는 2026년 기준, 컴퓨터는 쓰시지만 게임 개발은 처음 접해 보시는 분(특히 50대 이상 독자분)을 위해 작성했습니다. 이 레포지토리와 `CLAUDE.md`, `RULES.md`에 등장하는 낯선 단어들을 생활 속 비유와 함께 풀어 드립니다.

---

## 0. 큰 그림부터 한 번 보겠습니다

이 프로젝트는 **"컴퓨터 게임을 만드는 작업실"** 이라고 생각하시면 됩니다. 작업실에는 세 가지 주인공이 있습니다.

1. **유니티(Unity)** — 게임을 만드는 공장이자 도구함입니다.
2. **사람 개발자** — 게임의 기획과 설계를 책임지는 사람입니다. 이 레포에서는 프로젝트 주인이십니다.
3. **Claude Code 에이전트** — 사람의 지시를 받아 실제 코드를 타이핑해 주는, 인공지능 조수입니다. 지금 이 문서를 써 드린 **저 자신(Claude)** 이 바로 그 에이전트입니다. 앞으로 이 문서에서 "에이전트" 라는 말이 나오면 **"저(Claude) 같은 AI 조수"** 를 가리킨다고 보시면 됩니다.

세 주인공이 함께 일하려면 **같은 언어, 같은 규칙**을 써야 합니다. 이 문서는 그 언어의 사전입니다.

## 1. 스킬(Skill)과 슬래시 명령(/명령)

이 문서를 읽다 보면 `/task-start`, `/task-done` 같은 **슬래시(/) 로 시작하는 이상한 명령어들**이 자주 등장합니다. 이것이 바로 **"스킬"** 입니다. 용어가 낯설어 보이시겠지만, 사실 옛날부터 온라인 게임을 해 오신 분들께는 아주 익숙한 개념입니다.

### 리니지·바람의 나라·월드 오브 워크래프트를 떠올려 보십시오

온라인 게임에서 캐릭터는 **여러 가지 스킬(기술)** 을 가지고 있습니다.

- 마법사라면 "파이어볼", "텔레포트", "리저렉션"
- 기사라면 "방패 강타", "도발", "성스러운 빛"

이 스킬들은 **미리 정해진 동작들을 한 방에 발동**시킵니다. 마법사가 "파이어볼" 버튼을 누르면, 화면에 불덩이가 생기고, 적에게 날아가고, 데미지를 입히는 일련의 과정이 **자동으로 순서대로** 일어납니다. 플레이어가 일일이 "불을 만들고, 방향을 정하고, 던지고, 맞추고..." 하고 지시할 필요가 없습니다.

**Claude Code 에이전트(= 지금 여러분과 대화 중인 저, Claude) 의 스킬도 똑같습니다.**

- 에이전트, 즉 **저 Claude 는 마법사처럼 여러 스킬을 익혀 둔 상태**입니다.
- 개발자가 `/task-start` 라고 입력하는 것은 마법사에게 **"파이어볼 써!"** 라고 외치는 것과 같습니다.
- 그러면 에이전트는 그 스킬에 적혀 있는 **정해진 순서(규칙 파일 읽기 → 파일 목록 확인 → 범위 선언)** 를 한 번에 실행합니다.

### 왜 슬래시(/)로 시작하나요

MMORPG에서 채팅창에 **`/파티`, `/귓속말`, `/친구`** 같은 명령어를 쳐서 기능을 쓰셨던 기억이 나실 겁니다. 스킬 이름 앞에 슬래시를 붙이면 **"이건 채팅이 아니라 시스템 명령입니다"** 라고 구분해 주는 전통입니다. Claude Code도 이 전통을 그대로 따릅니다.

### 이 프로젝트에 등록된 주요 스킬

마법사의 스킬북처럼, 에이전트가 쓸 수 있는 스킬 목록이 `.claude/skills/` 폴더에 정리되어 있습니다. 자주 쓰는 것만 먼저 소개해 드리면 이렇습니다.

| 스킬 이름 | 게임으로 비유하면 | 하는 일 |
|---|---|---|
| `/task-start` | "전투 준비" 버프 | 새 작업을 시작할 때, 규칙을 다시 확인하고 **수정할 파일 목록을 선언** 합니다. |
| `/task-done` | "승리 후 정리" | 작업이 끝나면 **결과 요약과 자체 검증** 을 한 번에 실행합니다. |
| `/self-update` | "스킬북 업데이트" | 이번에 배운 교훈을 **영구적인 규칙으로 승격**할지 검토합니다. |
| `/design` | "작전 회의" | 작업이 크면 **편집 구간을 미리 쪼개** 둡니다. |
| `/run` | "소환수 호출" | 유니티 에디터 같은 **외부 프로그램을 실행** 합니다. |

### 스킬이 없다면 어떤 일이 벌어지나요

마법사에게 스킬이 없으면 **주먹으로 싸워야 합니다**. 마찬가지로, Claude Code에 스킬이 없으면 개발자가 매번 **"먼저 규칙 파일을 읽어 주세요. 그 다음에 파일 목록을 확인해 주세요. 그 다음에..."** 하고 일일이 지시해야 합니다.

스킬을 한 번 잘 만들어 두면, 그 뒤로는 `/task-start` 한 마디로 같은 절차가 매번 정확하게 반복됩니다. **이것이 Claude Code가 일반 채팅 AI와 다른 이유**입니다.

## 2. 유니티(Unity)

세계에서 가장 널리 쓰이는 **게임 제작 엔진**입니다. "엔진"이라는 말이 어렵게 들리실 수 있는데, 자동차 엔진처럼 **게임이 돌아가는 데 필요한 기본 부품들(그래픽, 소리, 물리, 입력 장치 처리 등)을 한데 모아 둔 통합 도구**라고 이해하시면 됩니다.

- 쉬운 비유: 유니티는 "한글 프로그램" 같은 것입니다. 한글 프로그램이 글씨를 쓰고 표를 그리는 모든 기능을 제공하듯, 유니티는 게임 화면을 그리고 캐릭터를 움직이는 기능을 제공합니다.
- 이 프로젝트가 쓰는 버전은 **Unity 2022.3 LTS** 입니다. LTS는 "Long Term Support(장기 지원)"의 약자로, **3년 정도 안정적으로 유지 보수되는 믿을 만한 버전**을 뜻합니다.

## 3. 유니티 에디터(Editor)

유니티 프로그램을 실행했을 때 화면에 나타나는 **작업 창 전체**를 "에디터"라고 부릅니다. 한글 프로그램을 켜면 나오는 편집 창과 같은 것입니다. 개발자는 이 에디터 안에서 캐릭터를 배치하고, 버튼을 만들고, 게임을 시험해 봅니다.

- `/run editor` 라는 명령어가 있는데, 이것은 에이전트가 **"유니티 에디터를 켜 주세요"** 라고 할 때 쓰는 스킬입니다.

## 4. 에셋(Asset) — 흔히 "어셋"이라고도 부릅니다

게임에 들어가는 **모든 재료**를 통칭해서 에셋이라고 합니다.

- 캐릭터 그림, 배경 그림 → 이미지 에셋
- 효과음, 배경음악 → 오디오 에셋
- 캐릭터 3D 모델 → 모델 에셋
- 개발자가 짠 프로그램 코드 → 스크립트 에셋

이 모든 재료가 `Assets/` 라는 폴더 안에 모여 있습니다. 영어로 Assets라고 쓰기 때문에 우리말로 옮길 때는 "자산" 또는 그냥 "에셋/어셋"이라고 부릅니다.

- 쉬운 비유: 요리책에서 재료 목록과 같습니다. 밥을 지으려면 쌀, 물, 냄비가 필요하듯 게임을 만들려면 그림, 소리, 코드가 필요합니다. 이 재료들이 전부 에셋입니다.

## 5. 프리팹(Prefab)

프리팹은 영어 단어 Prefabricated(미리 조립된)에서 온 말입니다. 여러 에셋을 조합해 만든 **완성된 부품을 저장해 두는 틀**입니다.

- 예를 들어 "적 몬스터 한 마리" 를 만들려면 캐릭터 모델 + 체력 수치 + 움직이는 코드 + 공격 소리가 필요합니다. 이 네 가지를 묶어서 한 번 완성한 뒤 "몬스터 프리팹"으로 저장해 두면, 이후에는 **도장 찍듯이** 게임 안에 같은 몬스터를 수십 마리 복제해 놓을 수 있습니다.
- 쉬운 비유: 조립식 가구(이케아 같은)의 설계 도면과 부품 세트를 한 상자에 담아 둔 것과 같습니다. 한 번 만들어 놓으면 똑같은 가구를 몇 개든 찍어낼 수 있습니다.

## 6. 씬(Scene)과 게임오브젝트(GameObject)

- **씬(Scene)** 은 영화의 한 "장면"과 같습니다. "타이틀 화면 씬", "1단계 플레이 씬", "게임 오버 씬" 처럼 게임의 한 장면 단위입니다.
- **게임오브젝트(GameObject)** 는 씬 안에 놓인 **모든 물건** 입니다. 주인공, 몬스터, 배경 나무, 눈에 보이지 않는 소리를 내는 스피커 장치까지 전부 게임오브젝트입니다.

쉬운 비유: 영화 세트장을 떠올려 보십시오. 세트장 자체가 씬이고, 그 안에 놓인 의자, 배우, 카메라, 조명이 전부 게임오브젝트입니다.

## 7. 스크립트(Script)와 C#

스크립트는 **개발자가 쓴 짧은 글**이라는 뜻인데, 여기서는 **게임의 행동을 적어 둔 프로그램 코드**를 가리킵니다. 이 프로젝트는 **C#(씨샵)** 이라는 프로그래밍 언어를 씁니다.

- 쉬운 비유: 연극 대본과 같습니다. "주인공은 왼쪽 버튼을 누르면 왼쪽으로 걷는다", "체력이 0이 되면 쓰러진다" 같은 **행동 지침서**를 컴퓨터가 이해할 수 있는 말로 써 둔 것입니다.

## 8. MonoBehaviour (모노비헤이비어)

유니티에서 **살아 움직이는 물건의 대본**에 붙이는 특별한 이름표입니다. 개발자가 "이 스크립트는 유니티 게임 안에서 동작하는 코드입니다" 라고 알려 줄 때 씁니다.

- MonoBehaviour를 붙이면 유니티가 **매 순간**(정확히는 1초에 60번 정도) 이 스크립트를 자동으로 실행해 줍니다.
- 이 덕분에 "주인공이 계속 걷는다", "시간이 흐른다" 같은 연출이 가능해집니다.

## 9. 어셈블리(Assembly)와 .asmdef

스크립트가 늘어나면 **책처럼 묶어서** 관리해야 합니다. 이렇게 묶은 한 권이 "어셈블리"이고, 그 묶음의 이름표가 `.asmdef` 파일입니다.

이 프로젝트는 네 권의 책으로 나누어져 있습니다.

| 책 이름 | 무엇을 담나요 |
|---|---|
| `Project.Core` | 공통으로 쓰는 기초 도구 |
| `Project.UI` | 버튼, 메뉴 같은 화면 요소 |
| `Project.Combat` | 전투 관련 기능 |
| `Project.Rendering` | 화면에 그림 그리는 기능 |

- 왜 나누나요? **수정한 부분만 빠르게 다시 읽게** 하기 위해서입니다. 책이 한 권이면 글자 하나 고쳐도 책 전체를 다시 인쇄해야 하지만, 여러 권으로 나눠 두면 고친 책만 다시 찍으면 됩니다. 이 시간 절약이 개발 속도에 큰 차이를 만듭니다.

### autoReferenced = false 가 중요한 이유

`.asmdef` 파일 안에 `autoReferenced: false` 라는 설정이 있습니다. 이것을 **절대 `true`로 바꾸시면 안 됩니다**. 바꾸면 유니티가 모든 책을 한꺼번에 다시 읽어야 해서 속도가 크게 느려집니다. 이 규칙이 `RULES.md` 의 RULE-01 입니다.

## 10. 네임스페이스(Namespace)

한자로 옮기면 "이름 공간"입니다. 같은 이름의 물건이 겹치지 않도록 **주소처럼 앞에 붙이는 꼬리표**입니다.

- 예: `Project.UI.Inventory` 는 "Project라는 프로젝트의, UI라는 책의, Inventory(가방)"이라는 뜻입니다.
- 쉬운 비유: 서울특별시 종로구 세종대로 와 같습니다. 같은 "세종대로"가 다른 도시에 있을 수 있으니 앞에 "서울특별시"를 붙여 구분하는 것과 똑같습니다.

이 프로젝트는 **모든 코드에 `Project.` 로 시작하는 네임스페이스를 반드시** 붙이도록 약속하고 있습니다.

## 11. 플레이 모드(Play Mode)와 에디트 모드(Edit Mode)

유니티 에디터에는 재생(▶) 버튼이 있습니다.

- **에디트 모드(Edit Mode)** : 재생 버튼을 누르지 않은 상태. 개발자가 물건을 배치하고 코드를 고치는 "작업 중" 상태입니다.
- **플레이 모드(Play Mode)** : 재생 버튼을 누른 상태. 실제로 게임이 돌아가는 "시연 중" 상태입니다.

쉬운 비유: 연극의 리허설과 본 공연의 차이와 같습니다. 리허설(에디트 모드) 중에는 동선을 고치고 대본을 수정하지만, 본 공연(플레이 모드) 중에는 실제로 관객에게 보여 주는 움직임이 일어납니다.

## 12. Domain Reload (도메인 리로드)

영어로는 좀 낯선 말이지만, 풀어쓰면 **"재생 버튼을 누를 때마다 유니티가 모든 책을 처음부터 다시 읽는 동작"** 입니다. 기본 설정으로는 재생 버튼을 한 번 누를 때마다 약 2~10초씩 기다려야 합니다.

- 이 프로젝트는 이 기다림을 **꺼 놓았습니다**. 그래서 재생 버튼을 누르면 즉시 게임이 시작됩니다.
- 대신 **개발자가 지켜야 하는 규칙**이 생겼습니다. 한 번 켠 상태가 두 번째 재생에 남아 있으면 안 되는 변수들은, 개발자가 직접 "처음 상태로 되돌려라" 라고 지시해 줘야 합니다. 이 지시가 `[RuntimeInitializeOnLoadMethod]` 입니다.
- 비유하자면, 연극 리허설에서 세트장을 매번 새로 지을 돈이 없어서, 대신 배우들이 **매 리허설 시작 전에 소품을 제자리에 되돌려 놓는 약속**을 한 것입니다.

## 13. Rigidbody(리지드바디)와 물리 연산

Rigidbody는 "딱딱한 물체"라는 뜻인데, 게임오브젝트에 **중력과 충돌을 적용해 주는 부품**입니다.

- 예: 공에 리지드바디를 붙이면, 떨어뜨리면 바닥으로 굴러가고 벽에 부딪히면 튕깁니다.
- 이 물리 기능을 쓸 때는 **"지금 내가 공중에 떠 있나?", "지금 벽에 붙어 있나?"** 같은 상태를 먼저 확인한 뒤 힘을 가해야 합니다. 안 그러면 땅에 떨어지지 않은 캐릭터가 두 번 연속 점프해 버리는 등 버그가 생깁니다.

## 14. 컴파일(Compile)

사람이 쓴 C# 코드를 **컴퓨터가 이해하는 기계어로 번역**하는 과정입니다.

- 쉬운 비유: 한국어로 쓴 편지를 영어로 번역해서 미국 친구에게 보내는 일과 같습니다. 번역이 끝나야 상대가 내용을 알아봅니다.
- 코드를 저장할 때마다 유니티가 자동으로 컴파일을 돌립니다. 이때 문법이 틀리면 "컴파일 에러"가 뜹니다. 편지에 오탈자가 있어서 번역기가 멈춘 것과 같습니다.

## 15. 테스트(Test), EditMode와 PlayMode 테스트

게임이 잘 돌아가는지 자동으로 점검하는 **검사 프로그램** 입니다.

- **EditMode 테스트** : 재생 버튼을 누르지 않고도 돌릴 수 있는 짧은 검사입니다. 계산식이 맞는지 확인하는 정도입니다.
- **PlayMode 테스트** : 실제로 게임을 잠깐 돌려 보면서 하는 검사입니다. "주인공이 정말 점프하나?" 같은 동작을 확인합니다.

쉬운 비유: 자동차 정비소의 정기 점검과 시험 주행의 차이입니다.

## 16. 디버깅(Debugging)과 브레이크포인트(Breakpoint)

**버그(bug)** 는 코드에 숨은 오류이고, **디버깅**은 그 오류를 찾아 고치는 작업입니다.

- **브레이크포인트** 는 "여기서 잠깐 멈춰라" 라고 표시해 두는 **빨간 점** 입니다. 코드가 이 지점을 지날 때 게임이 일시 정지되고, 그 순간의 변수 값을 확인할 수 있습니다.
- 범인을 찾기 위해 감시 카메라 녹화를 특정 지점에서 잠깐 멈추고 자세히 들여다 보는 것과 비슷합니다.
- **DAP** 는 Debug Adapter Protocol의 약자인데, 그냥 "디버깅을 할 수 있게 해 주는 공용 규격" 정도로만 이해하셔도 충분합니다.

## 17. 깃(Git), 커밋(Commit), 브랜치(Branch)

**깃(Git)** 은 **코드의 변경 이력을 차곡차곡 기록해 두는 장부** 입니다.

- **커밋(Commit)** : 장부에 한 줄 적는 행위. "오늘 이 부분을 이렇게 고쳤습니다" 라고 도장을 찍는 것입니다.
- **브랜치(Branch)** : 가지라는 뜻으로, **장부를 일시적으로 둘로 갈라놓는 것**입니다. "메뉴 화면을 새로 만드는 작업"은 별도의 가지에서 하다가, 다 되면 원래 줄기에 합칩니다. 합치는 것을 **머지(merge)** 라고 합니다.

## 18. 워크트리(Worktree)와 심링크(Symlink)

이 프로젝트의 **조금 특별한 기능** 입니다.

- **워크트리** : 같은 프로젝트를 **여러 개의 작업 공간으로 동시에 열어 두는** 방법입니다. 에이전트 여러 명이 서로 다른 기능을 동시에 만들 때 서로 간섭하지 않기 위해 씁니다.
- 쉬운 비유: 같은 문서의 사본을 여러 벌 복사해서 각자 다른 방에서 수정하는 것과 비슷합니다. 나중에 모아서 합칩니다.
- **심링크(Symlink)** : "심볼릭 링크"의 줄임말로, **바로가기(shortcut)** 와 거의 같은 개념입니다. 워크트리 각 방에 무거운 그림 파일을 통째로 복사하지 않고, 원본을 가리키는 바로가기만 놓아 둡니다.

### 심링크 폴더는 수정 금지인 이유

바로가기를 고치면 **원본까지 바뀝니다**. 이 프로젝트에서 `Assets/Art`, `Assets/Prefabs` 같은 공용 폴더는 전부 바로가기이므로, 워크트리 안에서 이 폴더의 파일을 수정하면 다른 방의 작업까지 모두 흐트러집니다. 그래서 이런 폴더는 **"읽기만, 수정은 금지"** 로 약속되어 있습니다 (RULES.md RULE-02).

## 19. UPM (Unity Package Manager)

유니티에 기능을 **하나씩 묶음으로 추가해 주는 관리 도구** 입니다.

- 쉬운 비유: 휴대폰의 앱 스토어와 같습니다. 필요한 앱을 검색해서 설치하고, 필요 없어지면 지우는 것과 똑같은 방식으로 유니티에 새 기능을 붙였다 뗐다 합니다.

## 20. async / await / CancellationToken / UniTask

게임은 "천천히 시간이 흐르는 작업" 을 많이 다룹니다. 예를 들어 "5초 동안 체력이 서서히 차오른다" 같은 연출입니다. 이런 시간이 걸리는 작업을 다루는 문법이 `async` 와 `await` 입니다.

- 쉬운 비유: 세탁기를 돌리면서 다른 집안일을 하는 상황입니다. 세탁기가 돌아가는 동안(=async) 멍하니 기다리지 않고 설거지를 하다가, 세탁이 끝나기를 기다려(=await) 빨래를 넙니다.
- **CancellationToken** 은 **"이 작업을 중간에 멈추고 싶으면 이 버튼을 누르세요"** 라고 미리 달아 두는 **비상 정지 스위치** 입니다. 플레이어가 갑자기 화면을 닫았을 때 작업을 깔끔하게 멈추기 위해 필요합니다.
- **UniTask** 는 유니티에서 `async`/`await`를 더 가볍고 빠르게 쓰도록 해 주는 **외부 라이브러리** 입니다.

## 21. Claude Code와 에이전트(Agent)

**Claude Code** 는 Anthropic 이라는 회사에서 만든 **인공지능 개발 조수** 입니다. 터미널(명령창)에서 대화하듯 지시를 내리면, 코드를 대신 읽고 고쳐 줍니다.

- 이 프로젝트에서는 "에이전트" 라고도 부릅니다. 비서, 조수 정도로 이해하시면 됩니다. **지금 이 문서를 여러분께 써 드리고 있는 저 자신이 바로 Claude Code 에이전트입니다.**
- 에이전트에게 **"이렇게 일해 주세요"** 라는 지침서가 `CLAUDE.md`, `RULES.md` 이고, **"이 상황에는 이 절차를 밟으세요"** 라는 매뉴얼이 `.claude/skills/` 아래의 스킬들(1번 참고)입니다.

## 22. 전체 개발 사이클 — 한 작업의 흐름

에이전트와 함께 하나의 기능을 만드는 전체 흐름을 순서대로 써 드립니다.

1. **지시** — 개발자(사람)가 "이런 기능을 만들어 주십시오" 라고 자연어로 부탁합니다.
2. **`/task-start`** — 에이전트가 관련 규칙 파일을 다시 읽고, 건드릴 파일 목록을 선언합니다.
3. **설계(필요 시 `/design`)** — 편집 구간이 여럿이면 미리 쪼갭니다.
4. **코드 작성** — 에이전트가 선언한 범위 안에서만 파일을 수정합니다.
5. **컴파일 확인** — `./scripts/bridge-run.sh` 같은 도구로 **유니티를 띄우지 않고도** 문법 오류를 검사합니다.
6. **실행 검증** — 필요하면 유니티 에디터를 켜고 실제로 돌려 봅니다.
7. **테스트** — EditMode, PlayMode 테스트를 돌려 점검합니다.
8. **`/task-done`** — 변경 요약, 남은 청소, 검증 결과를 정리합니다.
9. **커밋** — 깃 장부에 한 줄을 기록합니다.
10. **`/self-update` 판단** — 이번에 얻은 교훈이 앞으로도 유용하면 규칙 파일에 반영을 제안합니다.
11. **머지(합치기)** — 브랜치를 본 줄기에 합칩니다.

이 사이클을 **하루에 여러 번** 반복하면서 게임이 조금씩 자라납니다.

## 23. 마지막으로 꼭 기억하실 네 가지

1. **`Assets/Scripts/` 가 비어 있어도 고장이 아닙니다.** 이 레포지토리는 "텅 빈 공책" 상태로 설계된 **템플릿** 입니다.
2. **`RULES.md` 의 규칙은 어겨서는 안 됩니다.** 어기면 프로젝트가 망가지는 불변의 약속입니다.
3. **심링크 폴더에 새 파일을 만들지 않습니다.** 새 프리팹이나 에셋은 그 기능을 쓰는 스크립트 바로 옆에 둡니다.
4. **에이전트를 의심하지 마시고, 묻지도 마시고, 확인은 하십시오.** 에이전트는 성실하지만 완벽하지 않습니다. 항상 변경된 내용을 직접 한 번 훑어 보시는 것이 가장 안전합니다.
